7 /*
8  * !!!!! IMPORTANT NOTE !!!!!:
9  * This is a C++ interface to the C library in R. C++ supports exceptions, but C
10  * does not. R's C library requires you to explicitly PROTECT() and UNPROTECT()
11  * memory for garbage collection. If care is not taken, and an exception is
12  * thrown after memory has been PROTECTed, but before it has been UNPROTECTed,
13  * UNPROTECT will not be called.
14  *
15  * Every effort should be made to not use exception-throwing functions between
16  * PROTECT and UNPROTECT pairs. If an exception-throwing function is used
17  * between pairs, a means of ensuring that UNPROTECT is called when exceptions
18  * are thrown is required.
19  *
20  * The ScopeGuard pattern could be used which would store protect_count
21  * internally, and call UNPROTECT(protect_count) in the destructor. It would be
22  * used as follows:
23  *
24  * guard = ProtectGuard();
25  * PROTECT(something);
26  * guard.increment_protect_count(1);
27  *
28  * ProtectGuard is an implementation of the ScopeGuard pattern. Its
29  * increment_protect_count method should be be called immediately after each set
30  * of PROTECT calls using the appropriate number of items to process. The
31  * ProtectGuard destructor will be called when the function exists, regardless
32  * of whether exceptions were thrown.
33  *
34  * For reference: the following methods are guaranteed to not throw exceptions:
35  * - std::unordered_map::begin()
36  * - std::unordered_map::end()
37  * - std::unordered_map::size()
38  * - std::unordered_map::operator[()
39  * - std::vector::begin()
40  * - std::vector::end()
41  * - std::vector::size()
42  * - std::vector::operator[()
43  */
